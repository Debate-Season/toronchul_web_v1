### 1.3 로그인 플로우 (실제 동작)

```
[로그인 버튼] 
  → https://kauth.kakao.com/oauth/authorize?client_id=...&redirect_uri=...&scope=openid
  → 카카오 인증
  → /oauth/callback/kakao?code=XXXX
  → (브라우저) POST https://kauth.kakao.com/oauth/token  (grant_type=authorization_code)
     ↳ { id_token, access_token, refresh_token }
  → (브라우저 → Next rewrite → 백엔드) POST /api/v2/users/login { socialType, idToken }
     ↳ { accessToken, refreshToken, profileStatus, termsStatus }
  → Zustand setTokens/setProfileStatus/setTermsStatus
  → router.push("/")
```

### 1.4 Apple 로그인 플로우 (코드 완료, 테스트 대기)

```
[Apple 버튼]
  → https://appleid.apple.com/auth/authorize?client_id=...&redirect_uri=...&response_mode=form_post
  → Apple 인증
  → POST /api/auth/apple/callback (Route Handler 수신)
     ↳ form body: { id_token, code, state }
  → Route Handler → GET /oauth/callback/apple?id_token=xxx&state=yyy 리다이렉트
  → (브라우저) state 검증 (sessionStorage)
  → (브라우저 → Next rewrite → 백엔드) POST /api/v2/users/login { socialType: "apple", idToken }
     ↳ { accessToken, refreshToken, profileStatus, termsStatus }
  → Zustand setTokens/setProfileStatus/setTermsStatus
  → router.push("/")
```

> **카카오와 다른 점**: Apple은 콜백에서 `id_token`을 바로 전달하므로 별도 토큰 교환 API 호출 불필요. 대신 `response_mode=form_post`라 Route Handler가 POST→GET 변환 담당.

### R-06 ✅ 콜백 에러 메시지 raw

**상태**: ✅ 수정 완료 (2026-04-06)

**수정 내용**: `toUserMessage()` 에러 코드별 사용자 친화 메시지 매핑 (`KOE320`, `ID_TOKEN_SIGNATURE_VALIDATION_FAILED` 등). 인가 코드 없이 직접 접근/새로고침 시 홈으로 리다이렉트. 카카오 에러 파라미터 감지. 이미 로그인 상태면 자동 홈 이동.

---

### R-07 🟢 Client Secret 설정 함정

**위치**: 코드가 아닌 카카오 콘솔 설정

**증상**: `exchangeCodeForIdToken` 이 `client_secret` 없이 호출. 카카오 콘솔 "Client Secret 사용 설정" 이 OFF 여야만 동작. 만약 보안 강화 목적으로 ON 전환 시 즉시 400. 운영 전환 시점 함정.

**해결안**: 운영 단계에서 Client Secret ON 으로 전환하기로 하면 교환 로직을 **서버 라우트(`src/app/api/auth/kakao/route.ts`)로 이동** — secret 은 `NEXT_PUBLIC_` 이 아닌 서버 전용 env 로 보관.

---

### R-08 🟢 `/api/v1/users/home` 500 과의 관계 명확화

현재 이 리포에는 `src/lib/api/home.ts:73-85` 에 "토큰 실은 호출 500 → 익명 호출로 1회 폴백" 패치가 들어가 있음 (2026-04-05). 이건 **카카오 로그인 자체 결함이 아니라 홈 API 의 개인화 경로 NPE** 로 판단됨 — 토큰이 있으면 서버가 유저 조인 시도 중 터짐. 따라서:

- **카카오 로그인 완료 시점 ≠ 홈 API 호환성 완료 시점**.
- 백엔드 팀에 "특정 토큰에서만 500, 익명은 200" 재현 사실 공유 필요. `bookMarks: null` 필드가 의심됨.
- 백엔드 수정 후 `home.ts` 의 fallback 블록은 제거.

## 5. 결정 기록 (ADR)

### ADR-01: client_id 를 `NEXT_PUBLIC_` 으로 노출
- **배경**: 카카오 REST API 키를 브라우저에서 OAuth authorize 호출에 사용해야 함.
- **결정**: `NEXT_PUBLIC_KAKAO_CLIENT_ID` 로 노출.
- **근거**: OAuth 2.0 에서 client_id 는 public credential 로 정의됨. 카카오 REST API 키 역시 이 용도로는 노출 허용 범위. 단, Admin API 엔드포인트 호출에는 사용 금지 — Admin API 키는 별도.
- **재평가 트리거**: 카카오 정책 변경 또는 Client Secret 전환 시 (R-07) 재검토.

### ADR-02: 코드 → id_token 교환을 브라우저에서 직접 수행
- **배경**: 서버리스 배포(Vercel) 환경에서 추가 서버 라우트 없이 빠르게 구현.
- **결정**: `kauth.kakao.com/oauth/token` 을 브라우저에서 직접 호출. `client_secret` 미사용.
- **근거**: 카카오가 해당 엔드포인트에 CORS 허용. public client 로 충분.
- **트레이드오프**: Client Secret 을 쓸 수 없음. 보안 강화 요구가 생기면 R-07 경로로 서버 라우트 도입.

### ADR-03: 로그인 API 만 `/api/v2` 사용
- **배경**: 대부분 API 가 `/api/v1/*` 인데 로그인만 `/api/v2/users/login`.
- **결정**: 백엔드 Swagger 결정을 그대로 따름.
- **검증 완료** (2026-04-06): Swagger 실제 경로와 DTO 확인됨. 프론트 DTO 와 완전 일치.

### ADR-04: 백엔드 audience 설정에 웹 REST API 키 추가 필요
- **배경**: 백엔드 `application.yml`의 `social.kakao.audience`에 모바일 네이티브 ��� 키(`18af8def...`)만 등록. 웹의 REST API 키(`eb083b8a...`)로 발급된 id_token은 `aud` 불일치로 거부됨. 에��� 코드는 `ID_TOKEN_SIGNATURE_VALIDATION_FAILED`��나 실제 원인은 audience 검증 실패.
- **결정**: 백엔드가 두 키 모두 허용하도록 수정 (2026-04-07 완료).
- **교훈**: 카카오 앱의 네이티브 앱 키 ≠ REST API 키. 같은 앱이라도 플랫폼별로 id_token의 `aud` 값이 다름.

### ADR-05: Apple Sign In `response_mode=form_post` 채택
- **배경**: 계획서(`docs/apple-login-plan.md`)에는 `response_mode=fragment`로 기술되었으나, Apple 웹 로그인 권장 방식은 `form_post`. `fragment`는 `useSearchParams()` 불가하여 수동 해시 파싱 필요.
- **결정**: `response_mode=form_post` 사용. Next.js Route Handler(`src/app/api/auth/apple/callback/route.ts`)가 Apple POST를 수신, query param으로 변환 후 `/oauth/callback/apple` 의 `page.tsx`로 GET 리다이렉트. (Next.js는 같은 디렉토리에 `route.ts`+`page.tsx` 공존 불가하므로 API 라우트로 분리.)
- **근거**: Apple 공식 권장 방식. Route Handler가 POST→GET 변환을 담당하므로 클라이언트 콜백 페이지는 카카오와 동일한 `useSearchParams()` 패턴 재사용 가능.
- **트레이드오프**: Route Handler 파일 1개 추가. id_token이 리다이렉트 query에 잠시 노출되나, 1회성 JWT이고 백엔드 audience 검증이 있으므로 수용 가능.
